================================================================================
                        STANDARD QUERY PATTERN
================================================================================

Every paginated method follows this pattern:

Step 1: Count Total (unfiltered by page)
    var totalCount = await _context.Entity
        .Where(e => !e.IsDeleted && ...filters...)
        .CountAsync();

Step 2: Get Paginated Records
    var items = await _context.Entity
        .Where(e => !e.IsDeleted && ...filters...)
        .OrderBy(e => e.SortField)
        .Skip(pagination.CalculateSkip())
        .Take(pagination.PageSize)
        .ToListAsync();

Step 3: Return Result with Metadata
    return PaginatedResult<Entity>.Create(items, totalCount, pagination);

This ensures:
- Accurate total count
- Efficient pagination using Skip/Take
- Consistent metadata calculation
- Clean separation of concerns

================================================================================
                        PERFORMANCE NOTES
================================================================================

✅ OPTIMIZED FOR PERFORMANCE
   - Skip/Take handled by database (not in-memory)
   - Where filters applied BEFORE pagination
   - Count operations separate from data retrieval
   - Include() relationships loaded with main query
   - AsNoTracking() can be used for read-only scenarios

⚠️  CONSIDERATIONS
   - Large COUNT queries on tables with millions of rows
   - Many INCLUDE statements can impact query complexity
   - Unindexed OrderBy columns may cause performance issues

💡 RECOMMENDATIONS
   - Add database indexes on OrderBy columns
   - Monitor query performance with profiler
   - Consider caching total counts if data is stable
   - Use max page size limit (100 items) to prevent abuse
